# Benchmark Design

## Benchmark Goal

The benchmark suite measures whether the MVP can detect known intent violations while avoiding blocking findings in clean or control projects.

The suite is intentionally small. It should be easy to understand, easy to extend, and deterministic enough to run in CI.

## Fixture Families

Fixtures live under `tests/evaluation/fixtures`.

| Fixture family | Purpose | Expected result |
| --- | --- | --- |
| `clean-project` | Verifies a conforming project has no findings. | Zero findings, zero blocking findings |
| `layer-boundary-violation` | Auth imports billing. | One PROVEN blocking finding |
| `forbidden-side-effect-violation` | Read-only report calls `save()`. | One PROBABLE non-blocking finding |
| `value-invariant-violation` | Discount literal exceeds max threshold. | One PROVEN blocking finding |
| `false-positive-cases` | Similar-looking but valid code. | Zero findings |
| `ambiguous-cases` | Heuristic signal requiring human review. | PROBABLE or SUSPECT, non-blocking |

## MVP Synthetic Dataset

The MVP benchmark should include a 50-case synthetic dataset:

- 15 layer boundary violations;
- 15 forbidden side effects;
- 10 value invariant violations;
- 10 clean cases.

Each case must carry metadata:

```yaml
case_id: broken-auth-import-001
expected_finding: true
expected_status: PROVEN
expected_rule_id: INV-AUTH-001
expected_severity: critical
should_block: true
```

The current executable version is generated by `tests/evaluation/synthetic-dataset-benchmark.test.ts`. Static fixture folders remain smaller and human-readable; the generated benchmark verifies the larger synthetic distribution.

## Test Files

| Test | Role |
| --- | --- |
| `tests/unit/*.test.ts` | Unit coverage for rule validation, registry behavior, import extraction, analyzer checks, and classifier blocking logic. |
| `tests/integration/*.test.ts` | Integration coverage for clean project analysis, broken project analysis, CLI JSON, CLI Markdown, and blocking exit behavior. |
| `precision-benchmark.test.ts` | Computes true positives divided by all findings and enforces precision >= 80%. |
| `recall-benchmark.test.ts` | Confirms known violation fixtures are detected. |
| `false-positive-benchmark.test.ts` | Confirms clean/control fixtures stay below the false-positive threshold. |
| `semantic-fixture-runner.test.ts` | Runs each fixture through the core pipeline and verifies expected statuses. |
| `synthetic-dataset-benchmark.test.ts` | Generates and runs 50 labeled synthetic cases across the MVP scenario distribution. |

## Ground Truth Format

Ground truth currently lives in `tests/evaluation/fixture-runner.ts` as explicit expected findings:

- fixture name;
- fixture directory;
- fixture kind;
- expected rule id;
- expected status;
- expected blocking behavior.

This is deliberately simple for MVP. If the fixture set grows, move the metadata into per-fixture JSON files.

## Precision Calculation

```text
precision = detected_expected_findings / total_findings
```

A finding is counted as expected when it matches the fixture's expected rule id, status, and blocking behavior.

## Recall Calculation

```text
recall = detected_expected_findings / expected_findings
```

The current recall benchmark focuses on known violation fixtures, not clean fixtures.

## False Positive Rate Calculation

```text
false_positive_rate = findings_on_control_fixtures / number_of_control_fixtures
```

For MVP, control fixtures are `clean-project` and `false-positive-cases`.

## PROVEN Finding Rate

```text
proven_finding_rate = proven_findings / total_findings
```

This should not be forced to 100%, because heuristic rule types can produce useful non-blocking `PROBABLE` findings. The important rule is that only approved critical `PROVEN` findings can block.

## Expansion Plan

Add fixtures in this order:

1. Promote representative generated synthetic cases into static fixtures when they reveal regressions.
2. More clean projects with realistic imports.
3. More false-positive controls around side-effect names.
4. Multiple violations in one file.
5. Candidate and deprecated rules that must not block.
6. Diff-based fixtures once CLI diff support lands.
7. Real anonymized partner repo excerpts, eventually 100-200 real-world inspired cases.

## CI Policy

Benchmark tests should run with `pnpm test`. They are not performance benchmarks; they are correctness and product-quality checks.
